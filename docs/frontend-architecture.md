# Frontend Architecture

## UX Brief
- Цель: зафиксировать стартовый стек и правила проекта так, чтобы следующий этап был не выбором технологий, а прямой 1:1 сборкой сайта.
- Базовая платформа: `Next.js App Router` как основа маршрутизации, SSR/SSG, metadata и дальнейшего масштабирования без смены фреймворка.
- Результат задачи: репозиторий должен содержать стартовый код, понятную структуру директорий и документ с неизменяемыми правилами по компонентам, ассетам, модалкам, адаптивам и медиа.
- Ограничение: репозиторий изначально пустой, значит архитектура должна быть не теорией, а рабочим стартовым шаблоном.

## UX States Matrix
| Зона | Стартовое состояние | Следующий этап |
| --- | --- | --- |
| Маршруты | `src/app` с App Router | Добавление страниц и route groups под макеты |
| Секции | `widgets/*` | Сборка page sections 1:1 по макету |
| Интерактив | `features/*` | Изоляция форм, модалок, меню, табов |
| Общий UI | `shared/ui/*` | Кнопки, контейнеры, typographic primitives |
| Контент/данные | локально в компонентах | перенос в `entities`/`shared/api` при появлении API или CMS |

## Visual Direction
- Базовый стек:
  - `Next.js` + `React` + `TypeScript strict`
  - `Tailwind CSS` для быстрой и точной layout-верстки
  - `CSS variables` в `globals.css` как слой токенов
  - `Radix UI` только для сложных доступных primitives, начиная с `Dialog`
  - `clsx` + `tailwind-merge` + `class-variance-authority` для управляемых вариантов компонентов
- Почему не чистый CSS/SCSS как основа:
  - Для 1:1 сайта важнее скорость точной компоновки секций и консистентность токенов.
  - Tailwind даёт быстрый layout, а CSS variables не дают превратить UI в хаотичный набор utility-классов.
- Почему не shadcn/ui целиком на старте:
  - Для проекта важнее собственный тонкий слой компонентов, чем массовая генерация шаблонов.
  - При необходимости можно точечно добавлять shadcn-паттерны поверх уже принятого стека.

## Component Contracts
- Слои проекта:
  - `src/app`:
    - маршруты, layout, metadata, глобальные стили.
  - `src/widgets`:
    - крупные секции страниц и композиционные блоки.
  - `src/features`:
    - интерактивные сценарии, привязанные к пользовательскому действию.
  - `src/entities`:
    - доменные сущности, карточки, модели контента. Добавляется по мере появления повторяемых сущностей.
  - `src/shared`:
    - ui-примитивы, конфиг, утилиты, токены, assets, lib.
- Правила компонентов:
  - `shared/ui` содержит только переиспользуемые примитивы без бизнес-смысла.
  - `widgets` не должны хранить сложное состояние; интерактив выносится в `features`.
  - Один компонент = одна ответственность. Секция страницы не должна одновременно быть и layout-контейнером, и формой, и модалкой.
  - Варианты компонентов оформляются через `cva` или явные typed props, а не через россыпь boolean-пропов.
- Подход к модалкам:
  - Единая база на `@radix-ui/react-dialog`.
  - Каждая модалка живёт как feature (`src/features/<name>/ui/...`).
  - Портал, overlay, focus trap, закрытие по `Esc`, lock scroll не пишутся вручную заново в каждом месте.
  - Контейнер модалки один, контент меняется через composition.
- Подход к адаптивам:
  - Mobile-first.
  - Контент проектируется от `320px`.
  - Основные контрольные точки: `md` для планшета, `xl` для широкого десктопа.
  - Адаптив строится не только сменой колонок, но и через изменение плотности, отступов, размеров media-blocks и typographic scale.
- Подход к медиаконтенту:
  - `public/`:
    - файлы, которые нужны по URL без импорта: favicon, og-images, static exports, видео, крупные фоны, self-hosted fonts.
  - `src/shared/assets/`:
    - SVG и локальные ассеты, которые импортируются в компоненты и должны участвовать в tree-shaking/typed imports.
  - Растровые изображения в интерфейсе рендерить через `next/image`.
  - Декоративные SVG:
    - либо inline как React-компоненты, либо как импортируемые локальные ассеты, если это часть UI-композиции.
  - Видео:
    - хранить в `public/video`, lazy-load, всегда предусматривать poster.
  - Шрифты:
    - хранить локально в `public/fonts` и подключать через `next/font/local`, а не через внешние CDN.

## Implementation
### Зафиксированный стартовый стек
1. `Next.js App Router` как основной runtime.
2. `TypeScript strict` как обязательный режим.
3. `Tailwind CSS` + `globals.css` с CSS variables для design tokens.
4. `Radix Dialog` для модалок.
5. `cva`, `clsx`, `tailwind-merge` для компонентных вариантов.

### Структура проекта
```text
src/
  app/
  widgets/
  features/
  entities/
  shared/
    assets/
    config/
    lib/
    ui/
public/
docs/
```

### На чём разворачивать стартовый код
- Разворачивать стартовый код на `Next.js` без промежуточных шаблонов и конструкторов.
- Команда запуска после установки зависимостей:

```bash
pnpm install
pnpm dev
```

### Правила для дальнейшей 1:1 сборки
- Сначала собираются layout-контуры и media-блоки секций.
- Затем типографика, сетка, отступы и токены.
- Затем интерактивные состояния, модалки и формы.
- Новые зависимости добавляются только если они закрывают системную задачу, а не один визуальный частный случай.
