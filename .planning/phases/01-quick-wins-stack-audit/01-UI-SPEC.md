## Spacing Scale

Declared values (multiples of 4). Project uses Figma-exact pixel values in Tailwind arbitrary classes, not a token system. The values below map to the 8-point scale used throughout the form:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (CheckboxCheckIcon `size-[14px]` internal) |
| sm | 8px | Not used in form scope |
| md | 12px | Gap between checkbox visual and label text (`gap-[12px]`) |
| lg | 16px | Below-1024 form card inner padding (`px-4`) |
| xl | 24px | Standard inner gap between field groups (`gap-6`), desktop form `px-6` |
| 2xl | 30px | 1440 form padding (`p-[30px]`), 1440 field gap (`gap-[30px]`) |
| 3xl | 40px | 1024 form bottom padding (`pb-9`) |

Exceptions:
- Checkbox touch target: `size-6` (24px) — matches WCAG 2.1 AA minimum
- Button height: `min-h-[59px]` (1440), `h-[56px]` (1024), `min-h-[52px]` (768), `min-h-[44px]` (480/360) — breakpoint-specific, not a simple token scale
- Form underline padding: `pb-[20px] pt-[10px]` — Figma-exact value, not on scale

**File to modify:** `src/widgets/lead-form/ui/lead-form-fields.tsx`

**Current state (lines 438–487):** The structure already uses the correct pattern — `<input type="checkbox">` outside the label with `id={consentId}`, and `<label htmlFor={consentId}>` wrapping the visual checkbox span and text span. The nested `<a href="/privacy" onClick={(e) => e.stopPropagation()}>` is already present at line 478.

**What to verify/fix:**
1. Confirm the `<input type="checkbox">` at line 439 has `id={consentId}` — it does.
2. Confirm `<label htmlFor={consentId}>` at line 449 — it does.
3. Confirm `e.stopPropagation()` on the `<a>` at line 481 — it does.
4. The `<a>` is missing `target="_blank"` and `rel="noopener noreferrer"` — **this must be added** (decision D-04).

**Required change — add to the `<a>` element at line 478:**
```tsx
target="_blank"

```

**Interaction contract:**
- Click on label text outside the link → toggles checkbox
- Click on "Политикой конфиденциальности" link text → opens `/privacy` in new tab, checkbox state unchanged
- Click on visual checkbox square → toggles checkbox (handled by `<input id>` + `<label htmlFor>` association)

**Accessibility:** The `<input type="checkbox">` is visually hidden (`sr-only`) but reachable by keyboard. The visual checkbox square is `aria-hidden="true"`. No changes needed to a11y structure.

### FORM-03 — Button Text Centering in Case Forms

**Files to modify:**
- `src/widgets/first-screen/ui/first-screen-consultation-modal-1440.tsx` (lines 501, 543)
- `src/widgets/first-screen/ui/first-screen-consultation-modal-1024.tsx`
- `src/widgets/first-screen/ui/first-screen-consultation-modal-768.tsx`
- `src/widgets/first-screen/ui/first-screen-consultation-modal-480.tsx`
- `src/widgets/first-screen/ui/first-screen-consultation-modal-360.tsx`

**Root cause:** Buttons have both `flex items-center justify-center` AND asymmetric vertical padding `pb-[20px] pt-[24px]`. The asymmetric padding shifts the flex content downward visually, fighting the `items-center` alignment. When a fixed height is also set (`h-[60px]`), the padding is redundant and causes miscentering.

**Fix approach (D-07 — Claude's discretion):** Remove `pb-[20px] pt-[24px]` from the button classes. Keep `flex items-center justify-center h-[60px]`. The flex centering alone is sufficient when no asymmetric padding overrides it.

**Contract:** After fix, "оставить заявку" text must be visually centered (equal whitespace above and below) within the button bounds at all breakpoints.

**Note:** The submit button in `lead-form-fields.tsx` (line 489–498) uses only `flex items-center justify-center` with no asymmetric padding — it is NOT affected by FORM-03 and must not be changed.
