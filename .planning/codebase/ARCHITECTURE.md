# Architecture

**Analysis Date:** 2026-04-22

## Pattern Overview

**Overall:** Feature-Sliced Design (FSD) — a layered frontend architecture where code is organized by domain slices, not technical concerns.

**Key Characteristics:**
- Monorepo single-app: Next.js App Router with Payload CMS co-located in the same process
- Two route groups: `(site)` for public-facing pages and `(payload)` for the CMS admin panel
- Viewport-driven rendering: every widget ships 5 separate UI components (360, 480, 768, 1024, 1440) selected at runtime by `useViewportLayout`
- Server Components fetch CMS data; Client Components handle interactivity and viewport branching
- Static fallback data embedded in `.data.ts` files — CMS errors never break rendering

## Layers

**`app/`:**
- Purpose: Next.js route definitions, layouts, metadata, robots/sitemap
- Location: `src/app/`
- Contains: Route files (`page.tsx`, `layout.tsx`), global CSS, favicon assets
- Depends on: `widgets`, `shared`
- Used by: Next.js router

**`widgets/`:**
- Purpose: Full page sections — the primary building blocks assembled in `page.tsx`
- Location: `src/widgets/`
- Contains: Each widget is a subdirectory with `index.ts`, `model/`, `ui/`, and optionally `lib/`
- Depends on: `shared`, `entities`, `features`
- Used by: `app/(site)/page.tsx`

**`features/`:**
- Purpose: Cross-cutting interactive behaviors not tied to a single section
- Location: `src/features/`
- Contains: `lead-modal/` — consultation modal demo pattern
- Depends on: `shared`
- Used by: `widgets`

**`entities/`:**
- Purpose: Domain model placeholder (currently empty, `.gitkeep` only)
- Location: `src/entities/`
- Contains: Reserved for future business entity types
- Depends on: `shared`
- Used by: `widgets`, `features`

**`shared/`:**
- Purpose: Reusable primitives with no domain awareness
- Location: `src/shared/`
- Contains: `ui/` (Button, Modal, Typography, Surface, Grid, etc.), `lib/` (hooks, utilities), `config/` (site constants)
- Depends on: nothing in `src/`
- Used by: all layers above

**`cms/`:**
- Purpose: Payload CMS configuration — collections, globals, admin UI components
- Location: `src/cms/`
- Contains: `collections/` (CasesAdvertising, CasesVertical, Media, SecretsPost, Users), `globals/` (SiteSettings), `components/` (custom admin fields)
- Depends on: Payload config types
- Used by: `src/payload.config.ts`, `app/(payload)/`

**`bones/`:**
- Purpose: Boneyard skeleton/shimmer animation JSON definitions generated by `boneyard-js`
- Location: `src/bones/`
- Contains: Per-breakpoint `.bones.json` files for cards; `registry.js` (auto-generated, registers all bones)
- Depends on: `boneyard-js`
- Used by: `shared/ui/boneyard-skeleton.tsx`, widgets that show loading skeletons

## Data Flow

**CMS-driven page render:**

1. `app/(site)/page.tsx` — async Server Component, `revalidate = 60`
2. Calls `getCasesForSite()` (`src/widgets/cases/lib/get-cases-for-site.ts`) and `getSiteSettings()` (`src/shared/lib/get-site-settings.ts`) in parallel via `Promise.all`
3. Both functions call `getPayload({ config })` to query SQLite via Payload ORM
4. On error or empty result, both return static fallback data from `*.data.ts` files
5. Props passed down to `<CasesSection>` and section visibility flags to other widgets
6. Client Components (`"use client"`) read `useViewportLayout()` to select the correct breakpoint variant to render

**Viewport branching:**

1. Widget root component (e.g., `CasesSection`) calls `useViewportLayout()` — returns `null` on SSR (avoids hydration mismatch)
2. Returns an empty placeholder `<div id="...section" />` when layout is `null`
3. On client, `resolveViewportLayout(window.innerWidth)` returns one of: `"360" | "480" | "768" | "1024" | "1440"`
4. The matching breakpoint component is rendered; all others render `null` (not hidden, not mounted)

**Modal / cross-section event bus:**

1. Any component calls `dispatchOpenConsultationModal(variant)` from `src/shared/lib/open-consultation-modal.ts`
2. `window.dispatchEvent(new CustomEvent("open-consultation-modal", { detail: { variant } }))` fires
3. `FirstScreen` listens via `useEffect` and opens the consultation modal with the requested variant

**Site load sequence:**

1. `SiteLoadOverlay` (`src/widgets/site-load/`) mounts immediately, blocks scroll
2. Preloads critical viewport assets (images + hero video) with a 4.5 s timeout
3. Runs logo animation (pill → camera → СОЛО lettermark assembly, ~2.8 s minimum)
4. Waits for React hydration (`appReady`) and critical assets (`criticalAssetsReady`)
5. Fades out and unmounts, restoring scroll

**CMS admin access:**

1. `app/(payload)/layout.tsx` calls `ensurePayloadSchema()` to auto-push DB schema if empty
2. Admin panel at `/admin` served by Payload's `RootLayout`
3. API routes at `/api/[...slug]` handled by Payload's catch-all route handler

## Key Abstractions

**`useViewportLayout` hook:**
- Purpose: Client-side breakpoint resolver returning one of 5 layout tokens
- Location: `src/shared/lib/use-viewport-layout.ts`
- Pattern: Returns `null` on initial SSR render; sets state in `useEffect` on client mount. All widgets guard on `null` to avoid layout flash.

**Widget `index.ts` barrel:**
- Purpose: Single public export surface per widget
- Examples: `src/widgets/cases/index.ts`, `src/widgets/footer/index.ts`
- Pattern: Re-exports named exports from `ui/` and `model/` subdirectories; no default exports

**Static data fallback:**
- Purpose: Every widget with CMS data has a `model/*.data.ts` with hardcoded 1440 design data
- Examples: `src/widgets/cases/model/cases.data.ts`, `src/widgets/footer/model/footer.data.ts`
- Pattern: `getCasesForSite()` returns static data on catch; widgets default-prop to static data

**`SiteSettings` Global:**
- Purpose: CMS-controlled feature flags for toggling entire page sections
- Location: `src/cms/globals/site-settings.ts`, consumed in `src/shared/lib/get-site-settings.ts`
- Pattern: Boolean flags (showCases, showTeam, etc.) passed from Server Component to each widget

**Breakpoint UI components:**
- Purpose: Each pixel-perfect Figma breakpoint is a separate React component
- Examples: `src/widgets/cases/ui/cases-section-1440.tsx`, `src/widgets/cases/ui/cases-section-360.tsx`
- Pattern: Named `<WidgetName><Breakpoint>` — e.g., `CasesSection1440`, `FooterSection480`

**Boneyard skeleton system:**
- Purpose: Shimmer loading skeletons generated from Figma measurements
- Location: `src/bones/`
- Pattern: `boneyard-js` CLI generates `.bones.json` per breakpoint; `registry.js` registers them; components use `<BoneyardSkeleton boneName="cases-vertical-card-1024" />`

## Entry Points

**Public site:**
- Location: `src/app/(site)/page.tsx`
- Triggers: Next.js request to `/`
- Responsibilities: Fetch CMS data (cases + site settings), conditionally render all page sections based on feature flags

**Site layout:**
- Location: `src/app/(site)/layout.tsx`
- Triggers: All routes under `(site)` group
- Responsibilities: HTML shell, Montserrat font loading, metadata base, `SiteLoadOverlay`, bones registry initialization

**CMS admin:**
- Location: `src/app/(payload)/admin/[[...segments]]/page.tsx`
- Triggers: Requests to `/admin/*`
- Responsibilities: Payload admin UI

**CMS API:**
- Location: `src/app/(payload)/api/[...slug]/route.ts`
- Triggers: API requests to `/api/*`
- Responsibilities: Payload REST and GraphQL API

**CMS preview:**
- Location: `src/app/(site)/preview/cases-advertising/[id]/page.tsx`, `src/app/(site)/preview/cases-vertical/[id]/page.tsx`
- Triggers: Payload live preview iframe
- Responsibilities: Render individual case cards/modals for editorial preview

**Instrumentation:**
- Location: `src/instrumentation.ts`
- Triggers: Next.js server startup in development
- Responsibilities: Auto-seeds Payload with static case data if collections are empty

## Error Handling

**Strategy:** Silent fallback — CMS errors never surface to the user; static data is always shown instead.

**Patterns:**
- `getCasesForSite()` wraps all Payload calls in `try/catch`, returns `casesVerticalCards1440` / `casesAdCards1440` on any error
- `getSiteSettings()` wraps Payload global fetch in `try/catch`, returns safe `DEFAULTS` (all sections visible, secrets hidden)
- `SiteLoadOverlay` uses a 4.5 s asset preload timeout so the overlay always dismisses even if resources fail
- `seedCasesFromStaticIfMissing()` called via `queueMicrotask` — failure is only `console.warn`

## Cross-Cutting Concerns

**Logging:** `console.warn` for non-critical background operations (seed, instrumentation). No structured logging library.

**Validation:** Payload field-level validation (`required`, `unique`) in collection configs. No separate validation library in the site layer.

**Authentication:** Payload built-in auth for CMS admin. Public site has no authentication. CMS collection `read` access is `() => true` (public read).

**SEO:** Metadata defined in `app/(site)/layout.tsx` (site-wide) and `app/(site)/page.tsx` (page-level). `sitemap.ts` and `robots.ts` at app root. Canonical URLs use `NEXT_PUBLIC_SITE_URL`.
