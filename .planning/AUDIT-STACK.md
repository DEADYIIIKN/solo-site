# Аудит стека: Next.js App Router vs React SPA

**Дата аудита:** 2026-04-22
**Автор:** Claude (плановый аудит Phase 1, AUDIT-01)

---

## Контекст

Сайт SOLO — это лендинг видеопродакшн-агентства (`soloproduction.pro`) с единственной публичной страницей. На странице расположены секции: первый экран, showreel, бизнес-цели, команда, кейсы, услуги, уровни, форма заявки и футер. Контент кейсов и настройки сайта управляются через Payload CMS 3 с SQLite-адаптером. Сайт задеплоен в Docker с Traefik.

Текущий стек: **Next.js 15.4.11 App Router + PayloadCMS 3.82.0 + React 19 + Tailwind CSS 4 + TypeScript**.

**Почему это решение важно.** Вопрос возник потому, что сайт фактически однопейджевый — нет клиентской навигации между страницами. Это типичный паттерн, при котором иногда переходят на чистый React SPA, избавляясь от «тяжести» Next.js. Данный документ фиксирует осознанное решение об отказе от миграции или её проведении.

---

## Сравнение

| Критерий | Next.js App Router | React SPA |
|---|---|---|
| **Интеграция с Payload CMS** | Нативная: `getPayload()` вызывается прямо в RSC-компонентах (Server Components). Payload и Next.js разворачиваются как единый монорепозиторий, shared types из `payload-types.ts` используются напрямую. | Требует превратить Payload в отдельный API-сервер (REST или GraphQL) с CORS, отдельным деплоем, разными URL. Shared TypeScript-типы перестают работать без дополнительного npm-пакета или monorepo-инфраструктуры. |
| **SEO / crawlability** | Сервер отдаёт полностью отрендеренный HTML — страница индексируется без проблем. Open Graph и Twitter-карточки генерируются server-side (Metadata API). | JS-бандл на клиенте, HTML-скелет пустой. Требует либо SSG/SSR-обёртки (Vite + vite-plugin-ssr, Astro и т.п.), либо принимает риски для SEO лендинга. |
| **ISR / ревалидация** | `export const revalidate = 60` в page.tsx — CMS-контент автоматически обновляется раз в 60 сек без полной пересборки. Встроенная функциональность Next.js. | ISR отсутствует. Альтернативы: (a) webhook-triggered rebuild при обновлении контента, (b) runtime fetch каждый визит (медленнее). Оба варианта требуют отдельной реализации. |
| **Размер клиентского бандла** | Более высокий overhead за счёт Next.js runtime (~45–60 KB gzip) и клиентской части RSC-протокола. Однако большинство компонентов — RSC (нет `"use client"` по умолчанию). | Потенциально меньший бандл при минимальных зависимостях, но бонус нивелируется при добавлении router, state manager, data-fetching библиотеки. Реальная разница несущественна для лендинга. |
| **Стоимость миграции** | 0 — текущий стек. | Высокая: переписать data fetching (убрать `getPayload()`, добавить REST/GraphQL клиент), отделить Payload от Next.js в отдельный сервис, настроить CORS, обновить Docker Compose (2 сервиса вместо 1), пересмотреть ISR-стратегию. Оценка: 2–4 рабочих дня только на инфраструктуру. |
| **Сложность деплоя** | Единый Docker-образ: Next.js + Payload + SQLite работают в одном процессе. Traefik проксирует один сервис. | Два отдельных сервиса (SPA bundle + Payload API). Раздельное масштабирование, отдельные health checks, CORS между ними, усложнённый Docker Compose. |
| **Developer Experience** | Текущая команда знакома со стеком. Turbopack hot reload работает. RSC отлаживаются в браузере. Нет overhead переключения контекста. | Необходимо освоить новый bundler (Vite), настроить routing (React Router или TanStack Router), организовать data layer с нуля. |
| **Долгосрочное обслуживание** | Next.js 15 — активно поддерживается Vercel, LTS-гарантии, крупное сообщество. Payload 3 синхронизирован с Next.js экосистемой (официальная интеграция). | React SPA без Next.js — зависит от выбора bundler/router. Vite + React Router стабильны, но интеграция с Payload требует custom поддержки. Дополнительный слой поддержки без очевидной пользы. |

## Recommendation

**Рекомендация: оставить Next.js App Router — миграция на React SPA нецелесообразна.**

**Обоснование:**

1. **Payload CMS — жёсткая зависимость от SSR.** Текущий page.tsx вызывает `getPayload()` напрямую в Server Component (`getCasesForSite()`, `getSiteSettings()`). Это не просто паттерн — это архитектурный фундамент. Миграция на SPA означает вынести Payload в отдельный API-сервис, написать REST/GraphQL клиент, настроить CORS и раздельный деплой. Это существенная инфраструктурная работа без какой-либо функциональной выгоды.

2. **ISR работает из коробки.** `revalidate = 60` обеспечивает свежесть контента кейсов и настроек без полной пересборки. В SPA это потребует отдельного решения.

3. **Сайт однопейджевый, но это не аргумент против Next.js.** Отсутствие клиентской навигации снижает ценность client-side router, но Next.js App Router отлично подходит и для однопейджевых сайтов — он не добавляет оверхед от неиспользуемого routing.

4. **Нулевая стоимость сохранения vs высокая стоимость миграции.** Действующая архитектура работает, команда её знает, деплой настроен. Миграция принесёт 2–4 дня инфраструктурной работы и новые риски — при нулевой пользователь видимой выгоде.

**Итог:** Next.js App Router остаётся правильным выбором для данного проекта на весь текущий milestone.
