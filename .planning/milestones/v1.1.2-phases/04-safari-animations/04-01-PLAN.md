<!-- The only API surface touched in this plan -->
From `motion/react` (package `motion@^12.38.0`, installed in Task 1):

```typescript
import { MotionConfig } from "motion/react";

// Props (subset actually used):
type MotionConfigProps = {
  reducedMotion?: "always" | "never" | "user";
  children: React.ReactNode;
};
```

- `reducedMotion="user"` — motion runtime checks `prefers-reduced-motion: reduce` media query and disables animations when the OS setting is on. This is the chosen default per D-11 (a11y) and RESEARCH.md §Open Questions #3.
- MotionConfig does NOT mount client-side providers that break SSR — it is a plain context wrapper that tree-shakes away when no descendant uses motion. Still, because it is a client-context, it must live inside a `"use client"` boundary OR the layout must render it as a client-boundary island.

Existing layout (`src/app/(site)/layout.tsx`) is a **server component** (no `"use client"`). To keep SSR of the layout server-rendered but provide MotionConfig, create a tiny client component `motion-config-provider.tsx` that renders `<MotionConfig reducedMotion="user">{children}</MotionConfig>` and import it into the server layout.

Rationale: the root layout `src/app/(site)/layout.tsx` is a server component (exports `metadata`, uses `localFont`). We cannot add `"use client"` to it — that would force every child tree through a client boundary and break `metadata` export. Instead we create a small client island that receives `children` and provides MotionConfig context. This matches Next.js App Router conventions for client-only context providers and keeps SSR untouched.

Per D-11 checklist item and RESEARCH.md §Open Questions #3: `reducedMotion="user"` is the safest default — motion will automatically skip animations when `prefers-reduced-motion: reduce` is set.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry → project | Supply-chain: adding a new dependency (`motion`) brings third-party code into the client bundle |
| Server render → client hydration | `MotionConfig` context crosses the `"use client"` boundary; mismatch would cause hydration errors |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | `motion` npm package (supply chain) | mitigate | Pin to `^12.38.0` (caret locks minor; lockfile pins exact resolution). `motion` is Framer-owned, 10M+ downloads/month, well-maintained per RESEARCH.md §Security Domain. Review lockfile diff in PR. |
| T-04-02 | Denial-of-service | Client-side animation runtime cost | mitigate | `<MotionConfig reducedMotion="user">` respects `prefers-reduced-motion` by default, disabling animations for users who signal OS-level preference. This caps animation cost for accessibility-sensitive users. |
| T-04-03 | Information disclosure | MotionConfig context leak | accept | MotionConfig carries no user data — only animation preference flags. No PII, no secrets. |
</threat_model>
